Project

A Project is a configuration scope within the payment gateway that defines how payments are processed and where events are delivered.

Each project represents a separate integration (e.g. a store, platform, or service) within a single gateway instance. It is used to group settings, credentials, and payment behavior under one logical unit.

All Invoices and Charges are always created within a project and follow its configuration.

Project Scope​

A project defines the operational context for payment processing, including:

  • API authentication credentials
  • Webhook delivery endpoint
  • Payment behavior rules
  • Invoice and charge processing settings

Project Settings​

Before creating invoices or charges, a project must be configured in the admin panel.

Available settings:

  • Project ID — identifier used for API authentication
  • Project Secret — secret key used for both API authentication and webhook signature verification
  • Webhook URL — endpoint for receiving payment events
  • Invoice Tolerance Percent — allowed deviation for marking an invoice as paid

Security Model​

The Project Secret is used in two ways:

  • API Authentication — validates incoming requests to the gateway
  • Webhook Verification — used by the merchant system to verify that incoming webhook events are genuinely sent by the gateway

This ensures secure communication in both directions between the merchant and the gateway.

Invoice Tolerance​

Invoice tolerance defines how strictly the paid amount must match the invoice amount.

Example:

  • Invoice amount: $100
  • Tolerance: 1%
  • Payment received: $99

In this case, the invoice is considered paid, since the deviation is within the allowed tolerance range.

Relationship with Core Entities​

All payment entities are created under a project:

  • Invoices inherit project configuration
  • Charges inherit project configuration
  • Webhooks are delivered according to project settings
  • API requests are authenticated using project credentials

Summary​

A project acts as a configuration unit that groups authentication, payment behavior, and event delivery rules for a specific integration.
