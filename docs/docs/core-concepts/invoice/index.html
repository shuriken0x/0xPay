Invoice

Invoices are the core payment primitive of the gateway. They represent a time-bound payment request tied to a specific token and blockchain network.

When an invoice is created, a unique payment address is generated. The payer is expected to send the required amount to this address within the invoice lifetime.

Each invoice contains a fixed fiat-denominated price defined by the merchant. The system continuously monitors incoming blockchain transactions and updates the invoice state based on detected payments and their confirmation status.

Invoices support multiple payment scenarios, including partial payments, overpayments, and expired invoices. Final invoice resolution is determined based on confirmed blockchain transactions or expiration rules.

How to create​

Request​

Request
curl -X POST "$ZERO_PAY_URL/api/invoice/create" \
  -u "$ZERO_PAY_PROJECT_ID:$ZERO_PAY_PROJECT_SECRET" \
  -H "Content-Type: application/json" \
  -d '{
    "token": "ETH",
    "blockchain": "ethereum",
    "fiat": {
      "currency": "USD",
      "price": 100
    },
    "payload": "123"
  }'
FieldTypeDescription
tokenstringCryptocurrency token used for payment (e.g. ETH, BTC).
blockchainstringBlockchain network where the payment will be processed (e.g. ethereum).
fiat.currencystringFiat currency code used for pricing (e.g. USD, EUR).
fiat.pricenumberInvoice amount in fiat currency defined by the merchant.
payloadstring | nullOptional merchant-defined identifier or metadata (e.g. order id, user id).

StatusDescription
pendingThe invoice has been created and is waiting for a payment.
paid-partiallyA payment was received, but the amount is lower than the required invoice amount.
paidThe invoice has been fully paid with the exact required amount.
overpaidThe received payment amount exceeds the required invoice amount.
expiredThe invoice is no longer active and is no longer monitoring incoming payments.

How to use​

To process a payment, the client application must extract the following fields from the invoice response:

  • token
  • blockchain
  • address
  • amount - [pricing.crypto.amount]
  • expiresAt

These values should be displayed to the user in the payment interface.

In most cases, it is recommended to also generate and display a QR code to simplify the payment process, especially for mobile wallets. The QR code should encode the full payment details (token, blockchain, address, and amount) in a format compatible with the target wallet or payment standard used by the integration.

Amount Formatting​

The amount field may contain a long fractional part depending on token precision. For UI display purposes, it is recommended to limit the number of decimal places (e.g. up to 8 digits or fewer).

When truncating the value, the last visible digit should be rounded upwards to compensate for the discarded fractional part. This ensures the displayed amount is never lower than the required payment amount due to precision loss, preventing underpayment.

Webhook​

Invoice status is updated only based on confirmed payment events (invoice:payment:confirmed) and the invoice:expired event.

Status updates are allowed only until the invoice reaches a final state: paid, overpaid, or expired.

Webhook Events​

EventDescription
invoice:payment:pendingA payment for the invoice has been detected and is awaiting blockchain confirmation.
invoice:payment:confirmedA payment for the invoice has been successfully confirmed on the blockchain.
invoice:payment:failedA detected invoice payment has failed or became invalid.
invoice:expiredThe invoice has expired and is no longer accepting or monitoring payments.

Webhook Payment Statuses​

StatusDescription
pendingThe payment has been detected on the blockchain and is awaiting final confirmation.
confirmedThe payment has received the required blockchain confirmations and is considered successful.
failedThe payment is considered invalid or unsuccessful. This may happen if the transaction is dropped, replaced, reverted, or otherwise becomes unusable.
