Overview

Key Features​

The payment gateway is designed to provide a fully self-contained, developer-friendly crypto payment infrastructure with minimal external dependencies and maximum control.

Core Principles​

  • No KYC / No AML
    The system does not require identity verification or compliance onboarding to start accepting payments.

  • Self-Hosted
    The entire infrastructure can be deployed on your own servers, giving full control over data, routing, and payment logic.

  • Source Code Access
    Full source code is available, allowing customization, auditing, and deep integration into existing systems.

  • No Third-Party APIs
    The gateway operates without external payment providers or custodial services, reducing dependency and external risk.

  • White-Label Architecture
    The gateway does not enforce any predefined checkout interface or payment flow. Merchants fully control how payments are displayed and integrated, including UI, checkout logic, and presentation of payment details such as addresses, amounts, and QR codes.

Payment Capabilities​

  • 10+ Supported Blockchains
    Multi-chain support across major ecosystems including EVM and non-EVM networks.

  • Project-Based Architecture
    The system is organized into separate projects. Each project has its own configuration, allowing isolation of payment logic, settings, and environments.

  • Reliable Webhooks
    Guaranteed delivery with retry mechanism, ordered event processing, and idempotency support.

  • Telegram Notifications
    Real-time payment and system alerts can be delivered via Telegram for operational monitoring.

Developer Experience​

  • Coinbase Commerce–style API
    Clean and minimal API design inspired by the original Coinbase Commerce interface for fast and simple integration.

  • Predictable Integration Flow
    Create invoice → display payment data → monitor webhook events → confirm payment status.

  • Designed for Automation
    Built for backend-first integrations with webhook-driven architecture and deterministic payment states.

Project​

A Project represents an isolated configuration scope within a single gateway instance.

Each project allows you to run multiple independent integrations (e.g. different stores, platforms, or services) under the same deployment, while keeping their settings and payment flows fully separated.

A project defines how payments are processed, where events are sent, and how invoices and charges behave within its scope.

Typical project configuration includes:

  • Webhook URL for event delivery
  • Payment behavior rules and limits
  • Access credentials (Project ID and Secret)

All invoices and charges are always created within a specific project and inherit its configuration.

👉 See full details in the Project section.

Core Entities​

The gateway is built around two primary payment primitives that define all payment flows: Invoice and Charge.

Invoice​

An Invoice is a time-bound payment request tied to a specific token, blockchain network, and fiat-denominated amount.

Each invoice generates a unique payment address and is tracked for a limited period of time. Payments are monitored only within the invoice lifetime, and its final state is determined based on detected and confirmed transactions.

👉 See full details in the Invoice section.

Charge​

A Charge is a persistent payment entity that continuously monitors a payment address for incoming transactions.

Unlike invoices, a charge does not expire and remains active indefinitely. It automatically detects and processes any supported token within the configured blockchain network, without being limited to a single asset.

This makes charges suitable for multi-asset receiving addresses, long-lived payment flows, and continuous balance accumulation under a single entity.

👉 See full details in the Charge section.

Forwarding (Automated Sweeping)​

The gateway includes an automated Forwarding mechanism that periodically consolidates user funds and transfers eligible balances to configured cold wallets.

Forwarding is used to reduce hot wallet exposure and improve operational security by sweeping funds based on configurable balance thresholds per token and network.

👉 See full details in the Forwarding section.

Architecture Overview​

The system is designed as a modular backend service with asynchronous processing where applicable (e.g. webhook delivery, payment tracking, background jobs). Redis is used for caching and short-lived state, while RabbitMQ handles reliable event-driven workflows.
